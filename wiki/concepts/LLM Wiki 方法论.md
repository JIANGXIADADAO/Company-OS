# LLM Wiki 方法论

> 一种用 LLM 增量构建持久化个人知识库的方法论。由 Andrej Karpathy 在 2026 年 4 月提出。

## 核心思想

不是 RAG（每次查询从零检索文档片段），而是让 LLM **增量构建和维护一个持久化的 wiki**。

- 知识编译一次，而非每次查询重新推导
- 交叉引用已经建立，矛盾已经标注，综合已经反映所有已读内容
- Wiki 随着每个来源和每个问题持续增值

## 三层架构

```
Raw Sources (不可变)
    ↓ LLM 读取
Wiki (LLM 全权维护)
    ↑ 指导
Schema/CLAUDE.md (行为规范)
```

### Raw Sources（原始来源）
- 用户策展的文档集合
- 不可变——LLM 只读不写
- 分类存放：文章、论文、书籍、播客、剪藏

### Wiki（知识库）
- LLM 生成和维护的 Markdown 文件
- 包含：实体页、概念页、来源摘要、对比分析、查询归档
- 页面间通过 Wiki-link 密集互联
- 用户阅读，LLM 编写

### Schema（行为规范）
- 即 `CLAUDE.md`
- 定义目录结构、页面约定、工作流
- 使 LLM 成为有纪律的 wiki 维护者而非通用聊天机器人
- 用户和 LLM 共同演进

## 三大操作

### Ingest（摄入）
每新来源 → 读 → 讨论 → 写摘要 → 更新相关页 → 更新索引 → 记日志。一个来源可能触及 10-15 个页面。

### Query（查询）
问问题 → 读索引 → 读相关页 → 综合回答（带引用）→ 好答案归档回 wiki。

### Lint（健康检查）
检查矛盾、过时内容、孤儿页面、缺失页面、缺失交叉引用、数据空白。建议修复方案。

## 关键设计决策

1. **Index.md 优先于搜索引擎** —— 中等规模（~100 来源、~数百页面）下，纯文本索引足够，避免引入 RAG 基础设施
2. **Log.md 可 grep** —— `## [YYYY-MM-DD]` 格式使日志可被 Unix 工具解析
3. **Obsidian 是 IDE** —— LLM 做编辑，用户在 Obsidian 中浏览结果、跟进图视图
4. **Git 是版本控制** —— 天然的版本历史、分支、协作

## index.md 是路由器，不是 Agent

在多 Agent 场景中，index.md 的角色被进一步显式化：**它就是记忆路由器本身**。它是一个始终加载的导航地图——Agent 目视导航即可，不需要一个单独的"记忆分配 Agent"来转发请求。

- **index.md** 回答"有什么"——所有记忆文件的名称 + 描述 + 分类
- **overview.md** 回答"意味着什么"——核心论点的压缩版
- 但 overview.md 本身是 index.md 里的一个条目——overview 是被路由的，不是路由器

Agent 查询流程：读 index.md（已在上下文中）→ 定位相关页面 → 按 wikilink 深入。没有"检索"步骤——index 本来就在那里。

参见 [[concepts/Agent 记忆分级系统]]。

## 适用范围

- 个人知识管理（目标、健康、心理学）
- 深度研究（论文、报告、论点演进）
- 阅读笔记（书籍章节 → 角色/主题/情节页面）
- 团队 wiki（Slack、会议记录、项目文档）
- 竞品分析、尽职调查、旅行规划、课程笔记、爱好深潜
- **多 Agent 记忆共享**（Cub 维护 wiki，Worker 只读消费，Inbox 模式自学习）

## 结构级变更的一致性

原始方法论假设变更主要来自**内容 intake**（新文章、新概念），频率低。但在公司 OS 场景中，**目录结构本身就是一个需要跨页面一致维护的数据实体**。解决方案：CLAUDE.md 的目录图是权威定义，结构变更后立即对"结构一致性"做定向检查。
