# CLAUDE.md 写作规范

> CLAUDE.md 不是规则堆——它是一个**按作用域分层的决策系统**。每一节定义了一类规则的作用范围和优先级。知道新规则该落在哪一节，就是避免"打补丁"的唯一方法。

## 节的优先级层级

CLAUDE.md 的节顺序不是随意的。它遵循一条认知流水线——从"启动时立刻要做什么"到"做完后怎么验证"：

```
启动即执行  → "立刻行动，无需判断"
硬约束      → "绝对不能违反的底线"
核心原则    → "做决策时的价值观"
目录结构    → "你在什么环境里操作"
页面约定    → "产出物长什么样"
工作流      → "具体怎么做"
格式模板    → "照着这个抄"
自检清单    → "做完了自查"
```

### 前两节：执行前必知

启动即执行 + 硬约束必须在任何推理发生之前加载。如果 Agent 先读了工作流开始执行，然后才看到约束说"你不能写 raw/"——已经晚了。边界必须先划定。

### 第三节：决策价值观

核心原则不是操作指令，是判断标准。当两条规则冲突时，用原则来裁决。它排在约束之后、操作之前——是约束的"为什么"，也是工作流的"指导思想"。

### 第四五节：操作环境

目录结构 + 页面约定是参考信息层。不是规则本身，是规则作用的对象。Agent 需要先知道文件系统长什么样、产出物什么格式，然后才能执行工作流。

### 第六七节：操作手册

工作流 = 具体步骤，格式模板 = 产出物规格。工作流引用模板，所以模板在后。

### 第八节：质量闸门

自检清单在所有操作完成后执行。它不定义规则，它验证规则是否被执行。排在最后，因为它是最后一道防线。

### 优先级冲突裁决

当两条规则冲突时，**排在前面的节获胜**：硬约束 > 核心原则 > 工作流步骤。如果工作流暗示可以修改 `raw/`，硬约束已经禁止了——不需要在工作流里重复。

## 如何判断新规则加在哪一节

不是按**添加时间**分类，是按**作用域**分类：

| 问题 | 答案指向 |
|------|---------|
| 每次启动时必须做的事？ | → **启动即执行** |
| 绝对不能违反的底线？ | → **硬约束** |
| 做判断时的价值观？ | → **核心原则** |
| 关于文件系统结构的事实声明？ | → **目录结构** |
| 关于 wiki 页面怎么写的规定？ | → **页面约定** |
| 某个操作的步骤级指令？ | → **对应的工作流** |
| 产出物的格式规范？ | → **格式模板** |
| 做完事后必须自查的验证项？ | → **自检清单** |

### 反例：什么构成"补丁式添加"

补丁的特征：**规则被触发场景驱动**（"我在改这个步骤，所以塞在这里"），而不是被规则的适用范围驱动。

- ❌ 把"必须创建 source 页"塞在 Ingest 步骤 3a 正文里 → 它应该是自检清单的一项
- ❌ 把"批量暂停规则"藏在步骤 2 的括号里 → 同上
- ❌ 把"结构一致性检查"放在工作流末尾当一句提醒 → 同上

补丁的共同后果：验证规则散落各处，未来想加新规则时，作者面临一个没有原则的选择。

### 正例：正确的归属

- ✅ "所有声明必须有来源引用" → **硬约束**（所有工作流通用）
- ✅ "每次 ingest 后必须更新 index.md" → **硬约束**（多个工作流触发，不是 ingest 独占）
- ✅ "知识编译一次，持续保鲜" → **核心原则**（价值观，不是操作指令）
- ✅ "批量摄入每 5 页暂停确认" → **自检清单**（验证项，不是步骤本身）

## 与记忆分级的关系

CLAUDE.md 是 Tier 0（宪法级）记忆。Agent 记忆分级系统要求宪法结构足够稳定，让 Agent 能形成可靠的行为锚点。如果每次加规则都改变节的顺序或命名，Agent 就无法可靠定位信息。"按作用域分类"之所以重要，正是因为作用域是稳定的，触发场景是随机的。
