# Agent 记忆分级系统

> **Agent 的记忆不是一块平板，而是按加载优先级、持久性和可变性分为五级的金字塔。**

## 五级记忆模型

```
        ┌─────────┐
        │ Tier 4  │  归档级：存但不主动加载
       ┌┴─────────┴┐
       │  Tier 3   │  过程级：会话内，用完即删
      ┌┴───────────┴┐
      │   Tier 2    │  领域级：按需按 wikilink 深入
     ┌┴─────────────┴┐
     │    Tier 1     │  地图级：每次会话必加载
    ┌┴───────────────┴┐
    │     Tier 0      │  宪法级：定义行为，极少变动
    └─────────────────┘
```

金字塔越往上，数量越多、加载频率越低、变化速度越快。

---

### Tier 0：宪法级（Constitution）

**内容**：CLAUDE.md

**特征**：
- 定义 Agent 的**行为规则**，而非知识
- 极少变动——改了就是升级了操作系统
- 所有 Agent 启动时**必定加载**
- 覆盖一切：任何决策如果违反宪法，就是越权

**不存什么**：不存事实、不存知识点、不存"谁是谁"。宪法只存"怎么做事"。

### Tier 1：地图级（Map / Router）

**内容**：index.md + overview.md

**特征**：
- **index.md** 是所有记忆文件的名称 + 一句话描述 + 分类目录。这是 Agent 的导航地图
- **overview.md** 是核心论点的压缩版——不是目录，而是"所有这些记忆合在一起意味着什么"
- 两者配合：index 回答"有什么"，overview 回答"意味着什么"
- 每次会话必加载，让 Agent 知道"我有哪些记忆可以用"而**不需要加载所有记忆**

**关键澄清**：index.md 是唯一的**路由器**——它是被索引的，overview.md 本身也是 index 里的**一个条目**。overview 是内容层的枢纽页面，不是基础设施。

### Tier 2：领域级（Domain）

**内容**：wiki/concepts/、wiki/entities/、wiki/sources/ 中的具体页面

**特征**：
- 数量最多、增长最快
- **按需加载**——Agent 从 Tier 1 的路由器知道有哪些页面，根据当前任务按 wikilink 深入
- 这是 LLM Wiki 方法论的核心价值：**知识编译一次，持续保鲜，不每次从零检索**
- entity 是实例（具体的人/产品/组织），concept 是抽象（方法论/框架/模式）
- Agent 的工作路径：**先读 concept 弄明白思路，再找 entity 拿现成工具**

**关键原则**：在多 Agent 场景中，Tier 2 对 Worker 是**只读**的。只有 Cub 有权写入——防止并发写入导致语义冲突。

### Tier 3：过程级（Procedural / Ephemeral）

**内容**：断点标记、当前任务状态、中间推理

**特征**：
- 只在当前会话内有效
- 跨会话时通过断点标记恢复，然后**立即删除标记**
- 零额外 I/O——断点存在必定被读取的地方（CLAUDE.md）
- 不与其他 Agent 共享——每个 Worker 的断点是自己的

**实例**：
```markdown
<!-- BUILDER:RESUME -->
**上次做到**：实现了核心模块
**下一步**：跑测试验证
<!-- /BUILDER:RESUME -->
```

### Tier 4：归档级（Archive）

**内容**：log.md 旧条目、已封存版本（v1/v2 历史）、旧复盘记录

**特征**：
- 不会主动加载——不在 index.md 的导航路径上
- 但**保留为只读参考**——需要回溯时可以被检索到
- 封存后的版本**不可修改**

## 设计原则

1. **加载优先级 = 启动成本**：Tier 0-1 的加载成本恒定，不管知识库多大。这是系统可扩展的基础
2. **路由器是文件，不是 Agent**：index.md 就是路由器。不需要一个单独的"记忆分配 Agent"
3. **过程记忆不留痕**：用完即删，不污染长期知识
4. **归档不是删除**：封存的知识仍然可检索——只是不会主动加载
