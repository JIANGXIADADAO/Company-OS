# 预索引 vs 运行时扫描

> AI 理解复杂信息空间的两种根本不同范式。不仅适用于代码——也适用于数据仓库、知识管理、任何需要"找到正确东西"的场景。

## 定义

### 运行时扫描（Runtime Scan）
AI 代理每次任务时从零探索：用 grep 搜索符号、用 glob 找文件、用 Read 读取源码。每次任务重复这一过程。

### 预索引（Pre-indexed）
提前将信息空间解析为结构化的知识图谱（符号关系、调用图、文件结构），代理直接查询图谱获取答案，无需扫描原始文件。

## 对比

| 维度 | 运行时扫描 | 预索引 |
|------|----------|--------|
| **每次任务成本** | 高（重复扫描） | 低（直接查询） |
| **首次成本** | 零 | 需一次性索引 |
| **大型信息空间** | 成本线性增长 | 成本几乎恒定 |
| **新鲜度** | 实时 | 依赖增量同步 |
| **Token 消耗** | 高（大量工具调用） | 低（精准返回） |
| **工具调用次数** | 多（grep/glob/Read） | 少（语义查询） |

## 关键实证

### 代码场景

CodeGraph 基准测试（7 个开源项目）：
- VS Code (~10k 文件)：预索引节省 73% token、72% 工具调用
- Gin (~150 文件)：预索引节省 23% token、19% 工具调用

规律：**代码库越大，预索引优势越明显。** 小项目（~100 文件）运行时扫描已经足够高效。

### 数据仓库场景（Anthropic 内部实验）

给数据分析 Agent 直接 grep 访问数千个 SQL 文件，准确率提升**不到 1 个百分点**。约 80% 的错误回答中，正确答案已经存在于语料库。"答案存在"与"Agent 现在答对了"之间**无相关性**。

**结论**：瓶颈不是 access（能不能访问到信息），是 **structure**（能不能把问题映射到正确的实体）。信息在那里，Agent 看到了，但仍用不上。

## 同类模式

这个范式对立出现在多个独立领域中：

| 领域 | 运行时查询 | 预索引/预编译 |
|------|----------|-------------|
| **知识管理** | RAG（每次检索片段） | LLM Wiki（增量构建 wiki） |
| **代码理解** | grep/glob/Read 扫描 | 预索引知识图谱 |
| **数据仓库查询** | Agent grep 原始 SQL 语料库 | Reference Docs + Semantic Layer |
| **数据库** | 全表扫描 | 索引查询 |
| **编译器** | 解释执行 | 编译为本机代码 |

本质是同一个权衡：**前期投入换来长期效率**。当查询次数远大于索引次数时，预索引一定是更优策略。

## 适用场景判断

- **适合预索引：** 信息空间稳定、查询频繁、团队协作、需要跨会话一致性、Agent 需要 resolve entity
- **适合运行时扫描：** 一次性任务、小型空间、频繁变动的原型、Agent 只需要找"某段文本是否提到 X"
