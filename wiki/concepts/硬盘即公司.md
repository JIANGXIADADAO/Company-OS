# 硬盘即公司

> **整个一人公司就是移动硬盘上的一个文件夹。拔掉硬盘，公司下班。插上另一台电脑，继续。**

这是去平台锁定的最终极体现——不是抽象的原则宣言，而是物理上可执行的部署方案。

## 原理

```
                      ┌──────────────────┐
                      │     移动硬盘      │
                      │                  │
    任何电脑 ←── USB ─→│  我的公司/       │──→ 另一台电脑
                      │    ├── wiki/     │
                      │    ├── projects/ │
                      │    ├── inboxes/  │
                      │    └── ...       │
                      │                  │
                      └──────────────────┘
```

不依赖任何云服务的**运行时可用性**。云端备份（如 GitHub Sync）是冗余层，不是基础设施。

## 关键设计决策

### 1. 零数据库，零安装脚本

没有 `npm install`、没有 `docker-compose up`、没有数据库初始化。所有数据是纯文本 Markdown 文件。文件管理器就是数据库浏览器。

### 2. 不依赖任何特定软件的运行时

- **Claude Code** 读写 wiki（Cub 和 Worker 的真正运行环境）——但 Claude Code 本身不在硬盘里，是每台电脑上的标准工具
- **Obsidian** 可视化浏览（人类的阅读器）——同样不在硬盘里，每台电脑单独安装
- 核心原则：**硬盘里存的只有你的数据和你的配置**，不存任何可执行程序

### 3. Git 版本控制

整个 `我的公司/` 就是一个 Git 仓库。每次交互自动 commit。插到另一台电脑前 push 到 GitHub（可选冗余备份），从那台电脑上 pull。或者不 push——硬盘本身就是主副本。

### 4. .claude/ 跟着硬盘走

Claude Code 程序本身在 C 盘（npm 全局安装），不跟着硬盘。但 `.claude/settings.json` 和 `.claude/skills/` 是**你的创作**——跟着仓库走。插到另一台电脑上 `cd` 到这个目录启动 `claude`，自动加载你的全部技能和配置。

### 5. 角色模板系统

没有"多 Agent 框架"——只有一个 Claude Code 程序，多窗口多开，各自读各自的 CLAUDE.md。同一个程序、不同的文件、不同的角色。

**流程**：

```
agents/templates/builder.template.md    ← 写一次，定义通用角色指令
        │
        └──→ 复制到 projects/new-project/CLAUDE.md
             │
             填 3 个占位符：{{PROJECT_NAME}} / {{PROJECT_DESC}} / {{TECH_STACK}}
             │
             cd 到该目录 → claude → Builder Worker 醒了
```

**模板的核心设计**：所有路径都是固定的——因为所有项目都嵌套在 `projects/` 下，`../../wiki/` 和 `../../inboxes/builder.md` 的路径永不变化。模板里只需三个占位符：项目名、描述、技术栈。

## 新电脑首次启动

```
1. 装 Obsidian（可选，仅用于人类浏览）
2. 装 Claude Code：npm install -g @anthropic-ai/claude-code
3. 认证：claude login（仅一次，需要联网）
4. 打开终端，cd 到仓库目录
5. 启动 Cub：claude
```

Cub 读到 CLAUDE.md、index.md、overview.md——所有记忆全部在位。

## 五层保障模型

```
Layer 5: 工具无关性  ←  不绑定 Obsidian，不绑定 Claude Code
Layer 4: 格式开放性  ←  Markdown 纯文本，任何编辑器可打开
Layer 3: 版本可追溯  ←  Git 记录每次变更
Layer 2: 质量强制力  ←  Cub 的强制引用
Layer 1: 物理所有权  ←  硬盘在你包里，不依赖任何云服务
```

第一层保障是物理的——硬盘在你包里。这是去平台锁定的地基。

## 优势与局限

**优势**：
- 零供应商锁定——公司物理上属于你
- 零网络依赖——飞机上一样工作（除了 API 调用本身需要网络）
- 复制即备份——拖拽文件夹到另一块硬盘 = 完整备份
- 零隐私风险——知识库不经过任何第三方服务器

**局限**：
- Claude Code 需要在每台电脑上安装（一次性）
- API 调用依赖网络（Claude API）
- 硬盘丢失 = 最后一次 Git push 之后的增量丢失（需要 GitHub 等远程作为冗余）
